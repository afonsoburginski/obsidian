# Modos de falha e recuperação

**Público:** quem provisiona ou opera o cluster. Consulte antes de subir o ambiente, para evitar as armadilhas conhecidas, e quando algo falhar, para diagnosticar e recuperar.

Este documento concentra num único lugar as armadilhas de instalação, os modos de falha em operação e os procedimentos de recuperação. Os demais documentos remetem a ele para não repetir o conteúdo. A maioria das armadilhas de instalação já está prevenida no chart ou no Terraform; a tabela registra o sintoma para diagnóstico rápido.

## 1. Armadilhas de instalação

| Sintoma                                                                         | Causa                                                                                                                                                       | Prevenção ou correção                                                                                                                 |
| ------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| Pods em `CrashLoopBackOff`; clientes Kafka com timeout                          | O Service `kafka` não expõe a porta interna 29092                                                                                                           | O Service deve expor 9092 e 29092; os clientes usam `kafka:29092`. Já configurado no chart.                                           |
| `cp-kafka` aborta no boot                                                       | O Service chamado `kafka` injeta a variável `KAFKA_PORT` (compatibilidade com Docker links), que o entrypoint do cp-kafka lê como configuração inválida     | `enableServiceLinks: false` em todos os pods. Já configurado no chart.                                                                |
| `ImagePullBackOff` nas imagens `atmanadmin/attlas-*`                            | O registro é privado e o CI não acessa o nó; as imagens não estão no containerd do worker                                                                   | Buildar com `npm run docker:build` e carregar com `load-images.sh`, uma vez por worker. Usar `imagePullPolicy: IfNotPresent`.         |
| Microsserviços não sobem ou recusam requisições autenticadas                    | `jwtSecret` ausente ou divergente entre os microsserviços e o gateway                                                                                       | Passar `--set jwtSecret=<valor>` no `helm install`, com o mesmo valor do `.env.docker` dos microsserviços. Não há default versionado. |
| `cp <ms>.env.docker` falha porque o arquivo não existe                          | Os `.env.docker` são ignorados pelo git; num clone limpo eles não existem                                                                                   | Gerar com `npm run setup:env` na raiz antes do deploy.                                                                                |
| Provider libvirt trava o `terraform init` numa versão incompatível              | O `.terraform.lock.hcl` foi versionado fixando uma versão incompatível do provider                                                                          | Não commitar o lock; o `versions.tf` já fixa o provider. Se ocorrer: `rm .terraform.lock.hcl && terraform init`.                      |
| `Could not open backing file: Permission denied` no `terraform apply`           | O libvirt ajusta a permissão do disco de topo para o qemu, mas não do backing file (a base compartilhada)                                                   | No laboratório: `security_driver = "none"` em `/etc/libvirt/qemu.conf` e reiniciar o libvirtd.                                        |
| `domain already exists` no `terraform apply`                                    | Domínio órfão de um apply anterior que falhou no meio                                                                                                       | `virsh undefine <nome>` e reaplicar.                                                                                                  |
| Kong em `CrashLoopBackOff` por falta de memória no boot                         | O Kong sobe um worker nginx por core do nó; em VMs grandes isso estoura o limite de memória                                                                 | `KONG_NGINX_WORKER_PROCESSES=2`. Já configurado no chart.                                                                             |
| Consumidores crasham no boot; o gateway responde 503 ("name resolution failed") | O `kafka-init` está desalinhado do catálogo `@attlas/contracts`: tópicos consumidos ou filas mortas (DLQ) não foram criados, e o consumidor aborta ao subir | Manter o `kafka-topics.list` alinhado ao catálogo de tópicos. Recuperação na seção 3.                                                 |
| O cluster aparece como `Unavailable` no Rancher                                 | O registro no Rancher é um passo separado do `terraform apply`; recriar o cluster por fora destrói o `cattle-cluster-agent` junto                           | Aplicar o manifesto de import após o `terraform apply`. Recuperação na seção 3.                                                       |
| Microsserviço crasha no boot logo após um deploy                                | O `.env.docker` do servidor é local e não é sincronizado pelo deploy; falta uma variável nova que o microsserviço passou a exigir                           | Editar o `.env.docker` do serviço no servidor e recriar o container. Recuperação na seção 3.                                          |

## 2. Modos de falha em operação

| Falha                                                             | Causa                                                                                                                                                                        | Mitigação                                                                                                                                                 |
| ----------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Um serviço não escala sob carga, mesmo com lag acumulado no Kafka | O scaler Kafka do KEDA não funciona neste ambiente (cp-kafka single-broker, listener `POD_IP`) e falha ao construir ("scaler with id not found")                             | Limitação conhecida do KEDA, não do chart. Os serviços escalam por CPU (o `ms-audit` já tem gatilho de CPU). Não depender apenas do gatilho de lag.       |
| O ScaledObject não fica `Ready` e o HPA não é criado              | Configuração de gatilho Kafka inválida: `sasl: none` é rejeitado, ou o `consumerGroup` difere do groupId real (`<serviço>-group-server`, com o sufixo que o NestJS adiciona) | Omitir `sasl`/`tls` (o Kafka interno não tem auth) e usar o groupId real. Já corrigido no chart.                                                          |
| Perda de dados ao reiniciar um pod de banco                       | No laboratório os bancos usam `emptyDir` (efêmero)                                                                                                                           | Produção usa Nutanix CSI, com volume persistente que reconecta quando o pod troca de nó. Nunca rodar produção com storage efêmero.                        |
| Pods sofrem OOM-kill ou eviction                                  | Memória não é compressível; a soma dos requests estourou a RAM do nó                                                                                                         | Dimensionar os requests para caber no nó. Ver [[02-DIMENSIONAMENTO]].                                                            |
| A perda de um nó derruba serviços em vez de reagendá-los          | A soma dos picos de réplicas não cabe em 2 dos 3 workers, violando o N+1                                                                                                     | Calibrar os tetos (`max`) para que o pico caiba em 2 workers. Ver [[02-DIMENSIONAMENTO]], seção 5.                               |
| O pool de disco enche e a escrita de auditoria e Kafka falha      | O banco de auditoria e o Kafka crescem até a retenção de 1 ano; o pool está subdimensionado ou não é expansível                                                              | Usar pool expansível, monitorar a ocupação e validar o tamanho com a taxa real de ingestão. Ver [[02-DIMENSIONAMENTO]], seção 8. |
| Backup inutilizável após uma falha de disco                       | O backup está no mesmo SSD dos dados vivos                                                                                                                                   | Manter o backup off-box (object storage ou segundo servidor).                                                                                             |

## 3. Recuperação

- **Tópicos Kafka faltando** (consumidores em crash, 503 no gateway): criar no broker os tópicos ausentes a partir do `kafka-topics.list` (incluindo as filas mortas `attlas.dlq.<tópico>`) e reiniciar os consumidores afetados.
- **Registro do Rancher quebrado**: remover o registro morto no Rancher e reaplicar o manifesto de import no nó server, para reinstalar o `cattle-cluster-agent`.
- **Microsserviço crashando por variável de ambiente faltando**: editar o `.env.docker` do serviço no servidor, adicionar a variável exigida e recriar o container do serviço.
- **Provider do Terraform travado**: `rm .terraform.lock.hcl && terraform init`, depois `terraform plan` para confirmar.
